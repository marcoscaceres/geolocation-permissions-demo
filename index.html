Runnable Examples (using mock APIs)

These use the same mock state machine as the panels above. Click a user state button, then run an example to see the result live.

Example 1: Approximate-only site (weather app)

// A weather app only needs city-level location.
// Under the single-permission model, this is all you need:
const status = await navigator.permissions.query({name: "geolocation"});
if (status.state === "granted" || status.state === "prompt") {
  navigator.geolocation.getCurrentPosition(
    pos => showWeather(pos.coords.latitude, pos.coords.longitude),
    err => showManualEntry(),
    { accuracyMode: "approximate" }
  );
}
// The site never asks about accuracy tiers. It declares what it needs
// and handles the result. Works identically under both models.

Example 2: Upgrade flow (ride-sharing app)

// Start with approximate (show nearby drivers),
// then upgrade to precise (pickup location).
//
// Step 1: Get approximate
navigator.geolocation.getCurrentPosition(
  pos => showNearbyDrivers(pos),
  err => showManualAddress(),
  { accuracyMode: "approximate" }
);

// Step 2: Later, when user taps "Set pickup location"
navigator.geolocation.getCurrentPosition(
  pos => setPrecisePickup(pos),
  err => handleError(err),
  { accuracyMode: "precise" }
);
// If only approximate was granted, this prompts. That's fine —
// the user tapped a button, they expect interaction.
// This is the SAME pattern as enableHighAccuracy today.

Example 3: The enableHighAccuracy precedent

// TODAY, this is already how the API works:
const status = await navigator.permissions.query({name: "geolocation"});
// status.state === "granted"

navigator.geolocation.getCurrentPosition(success, error, {
  enableHighAccuracy: true  // A HINT. No separate queryable permission.
});
// The UA might return low-accuracy position anyway (no GPS, indoor, etc.)
// There's no query({name: "geolocation-high-accuracy"}).
// There's no coords.wasHighAccuracy result attribute.
// The developer checks coords.accuracy and adapts.
//
// accuracyMode is architecturally the same kind of thing:
// a caller-side declaration of what accuracy the site needs.

Example 4: The privacy argument

// Two-permission model allows this ON PAGE LOAD, no user gesture:
const approx = await navigator.permissions.query({name: "geolocation-approximate"});
const precise = await navigator.permissions.query({name: "geolocation"});

if (approx.state === "granted" && precise.state === "prompt") {
  // SILENTLY learned: "user specifically chose approximate-only"
  // No prompt shown. No user awareness. Any 3rd-party script can do this.
  // This enables: nag UIs, A/B experiments on privacy-choosing users,
  // content gating ("precise location required for this feature").
  trackUserPrivacyPreference("approximate-only"); // dark pattern
}

// Single-permission model: only query({name: "geolocation"}) exists.
// Returns "granted" — site knows SOME access exists.
// Can't distinguish "user chose approximate" from "precise granted."
// The user's privacy preference is revealed only via coords.accuracy
// AFTER an active API call (user participated in the disclosure).
